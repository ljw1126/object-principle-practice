# 🏛️ 객체지향 설계 스타일 가이드 (오브젝트 - 조영호 저)

이 프로젝트는 조영호 님의 저서 '오브젝트'의 설계 철학을 준수합니다.

## 1. 책임 주도 설계 (Responsibility-Driven Design)
- **책임의 할당:** 객체는 스스로 처리할 수 있는 정보(데이터)를 기반으로 책임을 가져야 합니다. "정보 전문가" 패턴을 따르는지 확인하세요.
- **데이터보다 행동:** 객체를 상태(Data)가 아닌 협력에 참여하기 위한 행동(Message)의 집합으로 정의하세요.

## 2. 메시지와 협력 (Tell, Don't Ask)
- **묻지 말고 시켜라:** 다른 객체의 내부 상태를 Getter로 꺼내어 판단(Ask)하지 말고, 원하는 일을 하도록 메시지를 던지세요(Tell).
- **캡슐화:** 객체 내부의 세부 구현은 철저히 숨기고, 외부에 노출되는 메시지 인터페이스를 간결하게 유지하세요.

## 3. 낮은 결합도와 높은 응집도
- **추상화 의존:** 구체 클래스가 아닌 인터페이스나 추상 클래스에 의존(DIP 준수)하는지 확인하세요.
- **변경의 전파:** 하나의 클래스를 수정할 때 연쇄적인 수정이 발생하는지 파악하여 결합도를 낮추는 방향으로 제안하세요.

## 4. 유연한 설계 (SOLID)
- **OCP (개방-폐쇄 원칙):** 새로운 기능이 추가될 때 기존 코드를 수정하지 않고 확장이 가능한 구조인지 평가합니다.
- **상속보다는 합성:** 상속을 통한 재사용보다는 합성을 통한 유연한 협력을 우선시합니다.

## 5. JUnit 5 테스트 원칙
- 모든 비즈니스 로직은 적절한 단위 테스트를 포함해야 합니다.
- 테스트하기 어려운 코드는 대개 설계가 잘못되었을 가능성이 큽니다(높은 결합도 등). 테스트 용이성을 설계 지표로 삼으세요.
